# Project: Vision-Bridge MCP (VBMCP)
### 표준 MCP 기반의 비전-블라인드 코딩 모델을 위한 UI 디버깅 워크플로우

본 문서는 시각적 분석(Vision) 기능이 없거나 제한적인 LLM 모델이 UI/UX 개발(바이브 코딩 등) 과정에서 발생하는 화면 깨짐 및 레이아웃 오류를 스스로 인지하고 해결할 수 있도록, **MCP(Model Context Protocol)** 표준을 활용한 '스크린샷 촬영 및 외부 비전 에이전트 연동 파이프라인'을 구축하기 위한 명세서입니다.

---

## 1. 프로젝트 배경 및 목적

- **문제 상황 (The Pain Point):** - 최근 AI 에이전트를 활용한 실시간 코딩(Vibe Coding) 중 레이아웃이 깨지거나 스타일이 꼬이는 현상이 자주 발생함.
  - 비전 기능이 내장된 고성능 모델(예: Claude 3.5 Sonnet)은 화면을 캡처해 스스로 분석할 수 있지만, **비전 기능이 없는 모델(텍스트 전용 고속 모델, 로컬 LLM, 보안 특화 모델 등)**은 오로지 코드 구조(HTML/CSS/JSX)만으로 UI를 이해하려 하므로 시각적 버그를 디버깅하지 못함.
- **해결 방안 (The Solution):**
  - **MCP(Model Context Protocol)** 표준 규격을 도입하여 메인 코딩 모델에게 '눈(Vision)' 역할을 해줄 외부 도구를 쥐어줌.
  - 메인 모델이 필요성을 느끼거나 사용자가 명시할 때 **[스크린샷 촬영 MCP] -> [외부 비전 모델을 통한 분석 MCP] -> [텍스트 리포트 반환]** 과정을 거쳐 텍스트 전용 모델도 완벽하게 UI 버그를 인지하고 고칠 수 있는 멀티모달 코딩 워크플로우를 구현함.

---

## 2. 시스템 아키텍처 및 핵심 구성요소

시스템은 크게 **(1) 메인 코딩 에이전트(클라이언트)**, **(2) MCP 서버(도구 제공자)**, **(3) 외부 비전 분석 레이어** 3가지로 구성됩니다.


```

+-------------------------------------------------------------+
|                Main Coding Agent (Client)                   |
|         (Vision-Blind Text LLM or Local Model)             |
+------------------------------+------------------------------+
|
MCP 표준 프로토콜 통신
|
+------------------------------v------------------------------+
|                          MCP Server                         |
|  [Tool 1: take_screenshot]      [Tool 2: analyze_vision]     |
+--------------+------------------------------+---------------+
|                              |
(OS API 호출)                  (API / Ollama)
|                              |
+--------------v--------------+ +-------------v---------------+
|     Local OS / Browser      | | External/Local Vision Model |
|  (스크린샷 이미지 파일 생성)  | | (Claude Vision, Llava 등)   |
+-----------------------------+ +-----------------------------+

```

### ① 메인 코딩 에이전트 (MCP Client)
- 개발을 주도하는 메인 AI 모델 (예: Qwen, DeepSeek, 혹은 비전 기능이 꺼진 가벼운 로컬 모델 등).
- MCP 표준에 따라 제공되는 도구(`take_screenshot`, `analyze_vision`)의 명세를 주입받아, 사고 과정(CoT)에서 필요할 때 해당 도구를 자발적으로 호출함.

### ② 스크린샷 촬영 도구 (`take_screenshot`)
- **역할:** 현재 개발 중인 애플리케이션의 화면 또는 OS 전체 화면을 캡처하여 지정된 로컬 경로에 이미지로 저장.
- **구현 방식:** - 웹 앱 중심인 경우: `Puppeteer` 혹은 `Playwright` 기반의 MCP 서버 활용.
  - 일반 데스크톱/시뮬레이터인 경우: Python `pyautogui`나 `Pillow` 기반의 로컬 스크린샷 캡처 도구 구성.

### ③ 비전 분석 프록시 도구 (`analyze_vision`)
- **역할:** 저장된 스크린샷 이미지 경로와 프롬프트(예: "현재 UI에서 겹치거나 깨진 부분이 있는지 분석해줘")를 인자로 받음.
- **구현 방식:** - 비전 기능이 강력한 외부 API(Claude 3.5 Sonnet, GPT-4o) 혹은 로컬 비전 모델(Ollama 기반 Llama3.2-Vision 등)로 이미지를 전달.
  - 분석된 시각적 버그 정보를 정형화된 **텍스트 리포트(Markdown 또는 JSON)** 형태로 변환하여 메인 코딩 에이전트에게 반환.

## 3. 핵심 시나리오 & 워크플로우

1. **개발 진행 및 오류 발생:** 메인 코딩 에이전트가 프론트엔드 코드를 수정함. 브라우저 화면에서 버튼 위치가 밀리거나 텍스트가 가려지는 UI 깨짐 현상 발생.
2. **도구 호출 판단:** 사용자가 "UI 결과 확인해봐"라고 요청하거나, 에이전트 스스로 레이아웃 검증이 필요하다고 판단 시 `take_screenshot` 도구를 호출함.
3. **눈(Vision) 대행:** 촬영된 스크린샷의 경로를 받아 곧바로 `analyze_vision` 도구에 전달하며 디버깅을 요청함.
4. **시각 정보의 텍스트화:** 비전 모델이 이미지를 분석하여 다음과 같은 피드백을 텍스트로 넘김:
   > *"분석 결과: 좌측 사이드바가 접힐 때 메인 콘텐츠 영역의 여백(Margin)이 자동으로 줄어들지 않아, 우측 제출 버튼이 화면 밖으로 잘려 보입니다."*
5. **디버깅 완료:** 메인 코딩 에이전트는 제공받은 피드백(텍스트)을 분석하여 버그의 원인이 되는 CSS/Style 코드를 정확하게 찾아 수정 작업을 완수함.

## 4. 메인 에이전트 주입용 시스템 프롬프트 가이드

메인 코딩 AI가 이 워크플로우를 완벽히 숙지하도록 아래와 같은 지침(System Prompt)을 부여합니다.

```markdown
너는 UI/UX 수정을 담당하는 숙련된 코딩 에이전트이다. 
단, 너는 시각적 정보(이미지)를 직접 볼 수 없는 제약이 있다. 

따라서 프론트엔드 코드를 수정하거나, 사용자가 UI 관련 오류를 제보했을 때는 반드시 너에게 주어진 MCP 도구를 활용하여 '눈'을 대신할 정보를 확보해야 한다.

[행동 지침]
1. UI 컴포넌트의 렌더링 결과나 레이아웃을 확인해야 할 때는 주저 없이 `take_screenshot` 도구를 호출하라.
2. 스크린샷이 저장되면, 해당 파일 경로를 `analyze_vision` 도구에 넘겨 현재 화면 상황에 대한 상세한 시각적 리포트를 받아라.
3. 비전 분석 도구가 반환한 텍스트 리포트(깨짐, 정렬 불량 등)를 사람이 눈으로 본 것처럼 신뢰하고, 이를 바탕으로 코드를 디버깅하라.
4. 절대 눈이 없는 상태에서 대략적인 추측만으로 UI 버그 수정을 끝마치지 말라.

```

## 5. 다음 단계 (Next Action Items)

Claude와 함께 구체적인 코드를 작성하고 환경을 세팅하기 위해 아래 순서로 논의를 전개합니다.

1. **환경 정의:** 타겟 애플리케이션 형태(React 웹 앱, 모바일 에뮬레이터, 로컬 데스크톱 창 등)에 따른 스크린샷 도구 라이브러리 선정.
2. **MCP 서버 아키텍처 설계:** FastMCP(Python) 또는 TypeScript SDK를 활용한 커스텀 MCP 서버 베이스코드 작성.
3. **비전 모델 API 연동:** `analyze_vision` 내부에서 외부 API(Anthropic/OpenAI) 또는 로컬 Ollama 모델을 호출하는 프록시 로직 구현.
4. **클라이언트 테스트:** Cursor, Windsurf 또는 독립형 에이전트 환경에 본 MCP 서버를 등록하고 전체 루프가 작동하는지 검증.
